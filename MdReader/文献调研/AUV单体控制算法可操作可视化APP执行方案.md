# AUV 单体控制算法可操作可视化 APP 执行方案

## 1. 项目定位

本项目不是以科研论文对比为主要目标，而是面向横向项目交付展示。最终需要向甲方清楚展示：

1. AUV 单体控制算法模块已经搭建完成；
2. 控制算法链路能够运行；
3. 典型路径跟随任务可以演示；
4. 仿真结果可以可视化查看；
5. 展示系统可以在断网环境下运行；
6. 后续具备扩展为桌面软件或三维展示的基础。

因此，可视化系统的核心目标不是“论文式算法性能对比”，而是“工程交付展示 APP”。

## 2. 最终展示目标

最终希望形成一个可操作的本地展示 APP，甲方可以通过浏览器或桌面程序看到：

```text
项目首页
  -> 模块能力说明
  -> 控制算法流程
  -> 场景选择
  -> 仿真结果动画回放
  -> 状态曲线与控制输入
  -> 运行状态和结论说明
```

展示时应突出：

1. 该项目完成了 AUV 单体控制算法模块；
2. 模块包含路径制导、航向控制、速度控制、执行器限幅、仿真模型和数据输出；
3. 三类典型路径任务可以运行和展示；
4. 轨迹、误差、速度、控制输入可以同步查看；
5. 整个展示系统可以离线运行。

## 3. 推荐技术路线

综合展示效果、开发成本、离线能力和后续扩展性，推荐采用以下主路线：

```text
C++ 核心仿真
  -> CSV / JSON 数据输出
  -> React + Vite + ECharts 可视化 Web APP
  -> 本地静态打包离线展示
  -> 后续可选 Electron 封装为桌面 APP
```

### 3.1 核心仿真层 

使用工具：

```text
C++
CMake
Eigen
```

职责：

1. 保持现有 AUV 动力学模型；
2. 保持 LOS 制导和 PID 控制模块；
3. 生成标准化仿真数据；
4. 保持核心算法与展示界面解耦。

### 3.2 数据层

使用格式：

```text
CSV
JSON
```

第一阶段可继续使用 CSV，后续 Web APP 更适合读取 JSON。

建议数据流：

```text
C++ 仿真输出 CSV
  -> Python 脚本转换为 JSON
  -> Web APP 读取 JSON
```

这样做的好处：

1. C++ 端继续保持简单；
2. Web 前端读取 JSON 更方便；
3. 后续可以加入场景配置、summary 指标、模块信息等结构化数据。

### 3.3 展示 APP 层

使用工具：

```text
Node.js
React
Vite
ECharts
CSS
```

职责：

1. 提供项目首页；
2. 提供场景选择；
3. 提供 2D 动画回放；
4. 展示轨迹、误差、速度、控制输入；
5. 展示项目模块组成和运行状态；
6. 支持离线静态部署。

### 3.4 后续桌面封装层

可选工具：

```text
Electron
```

职责：

1. 将 Web APP 封装成 Windows 桌面程序；
2. 甲方可双击运行；
3. 减少命令行操作；
4. 提升交付感。

## 4. 离线展示要求

为了保证断网环境下也能展示，必须满足：

1. 不使用在线 CDN；
2. ECharts、JS、CSS 全部打包到本地；
3. 演示数据提前生成并放入本地目录；
4. 图片、图标、字体等资源全部本地化；
5. 打包结果可通过本地文件或本地服务打开；
6. 后续如封装 Electron，则可直接双击运行。

推荐离线打包方式：

```powershell
npm run build
```

生成：

```text
web/dist/
```

若浏览器不允许直接读取本地文件，可启动本地服务：

```powershell
cd web/dist
py -m http.server 8000
```

然后打开：

```text
http://127.0.0.1:8000
```

这仍然是离线运行，因为所有访问都发生在本机。

## 5. APP 功能设计

### 5.1 首页

首页用于向甲方说明项目完成内容。

建议包含：

1. 项目名称；
2. 项目定位；
3. 已完成模块清单；
4. 控制算法流程图；
5. 三个演示场景入口；
6. 离线运行说明。

首页重点展示：

```text
AUV 单体控制算法模块及可视化系统
```

而不是只展示误差曲线。

### 5.2 模块能力区

建议用卡片展示已完成模块：

```text
3-DOF AUV 仿真模型
LOS 路径制导模块
航向 PID 控制模块
纵向速度控制模块
执行器限幅模块
恒定海流扰动模块
CSV / JSON 数据输出模块
可视化展示模块
```

每个模块用一句话说明作用。

### 5.3 控制流程图

建议展示如下流程：

```text
参考路径
  -> LOS 制导模块
  -> 期望航向
  -> 航向 PID 控制
  -> 偏航力矩 N
  -> 速度控制
  -> 纵向推力 X
  -> AUV 3-DOF 仿真模型
  -> 状态输出
  -> 可视化展示
```

该图是交付展示中的关键内容，用于说明合同中的“控制算法模块”具体做了什么。

### 5.4 场景选择

第一版展示三个固定场景：

```text
演示 1：直线路径跟随能力
演示 2：圆形路径跟随能力
演示 3：S 形路径跟随能力
```

每个场景都应有：

1. 场景说明；
2. 演示目的；
3. 运行状态；
4. 轨迹动画；
5. 曲线图；
6. 简短结论。

### 5.5 2D 动画回放

每个场景页面应提供动画功能：

```text
播放
暂停
重置
时间滑条
播放速度选择
```

动画内容：

1. AUV 位置随时间移动；
2. AUV 航向箭头随时间旋转；
3. 参考路径保持显示；
4. 当前最近路径点或当前轨迹点可高亮；
5. 当前时间、位置、航向、速度、误差同步显示。

第一版动画建议使用 2D，不建议一开始做复杂 3D。

### 5.6 状态面板

动画旁边放实时状态面板：

```text
当前时间
x 位置
y 位置
航向角
纵向速度
横向误差
航向误差
纵向推力 X
偏航力矩 N
```

该面板用于让甲方看到系统不是单纯播放视频，而是基于仿真数据进行状态回放。

### 5.7 曲线图区

保留并优化当前已有图表：

```text
轨迹图
横向误差图
航向误差图
速度图
控制输入图
```

要求：

1. 每幅图有标题；
2. 每幅图有简短说明；
3. 坐标轴有名称、单位、刻度；
4. 控制输入图显示限幅线；
5. 图表与动画时间可以联动是后续增强项。

### 5.8 运行状态与结论

每个场景增加工程化结论，而不是科研式对比结论。

示例：

```text
该演示表明：AUV 单体控制模块能够在直线路径任务中稳定生成控制输入，并使航行器逐渐贴近参考路径。
```

圆形路径示例：

```text
该演示表明：模块能够处理连续转向路径，控制输出保持在限幅范围内，轨迹能够围绕目标路径运行。
```

S 形路径示例：

```text
该演示表明：模块能够处理方向连续变化的机动路径，具备基础路径跟随展示能力。
```

## 8. 分阶段实施计划

### 阶段 1：数据整理与 JSON 转换

目标：

```text
将当前 CSV 数据转换为 Web APP 易读取的 JSON 数据
```

任务：

1. 编写 CSV 到 JSON 转换脚本；
2. 固定 JSON 字段；
3. 为直线、圆形、S 形生成三个 JSON 文件；
4. 保留 summary 指标。

验证：

1. JSON 可以被浏览器读取；
2. 三个场景数据完整；
3. 字段单位和含义清楚。

### 阶段 2：搭建 React + Vite APP

目标：

```text
建立可运行的 Web APP 框架
```

任务：

1. 创建 `web/` 前端工程；
2. 添加首页；
3. 添加场景选择；
4. 读取本地 JSON 数据；
5. 显示基础轨迹图。

验证：

1. `npm run dev` 可以打开页面；
2. 页面可以切换三个场景；
3. 图表能显示对应数据。

### 阶段 3：实现 2D 动画回放

目标：

```text
让 AUV 在路径图上随时间运动
```

任务：

1. 添加播放、暂停、重置按钮；
2. 添加时间滑条；
3. 根据当前时间更新 AUV 位置；
4. 根据航向角绘制 AUV 朝向箭头；
5. 同步更新状态面板。

验证：

1. 动画播放流畅；
2. 时间滑条可拖动；
3. AUV 位置和曲线数据一致；
4. 航向箭头方向合理。

### 阶段 4：完善图表和工程说明

目标：

```text
形成适合甲方演示的完整页面
```

任务：

1. 增加模块卡片；
2. 增加控制流程图；
3. 增加 summary 指标；
4. 增加运行状态标签；
5. 增加每个场景的结论说明；
6. 美化页面布局。

验证：

1. 首页能说明项目完成内容；
2. 每个场景能说明演示目的；
3. 图表清晰；
4. 页面适合投屏展示。

### 阶段 5：离线打包

目标：

```text
确保断网环境下可以展示
```

任务：

1. 执行 `npm run build`；
2. 检查 `web/dist/`；
3. 断网打开本地页面；
4. 如有本地文件读取限制，使用本地服务；
5. 更新 README。

验证：

1. 不联网也可以打开；
2. 三个场景都能播放；
3. 图表和动画正常显示；
4. 所有数据来自本地文件。

### 阶段 6：可选 Electron 封装

目标：

```text
形成可双击运行的 Windows 桌面 APP
```

任务：

1. 添加 Electron；
2. 加载 `web/dist/`；
3. 打包 Windows 可执行程序；
4. 测试双击运行；
5. 整理交付说明。

验证：

1. 双击 exe 能打开 APP；
2. 不需要手动启动服务；
3. 数据和图表正常显示；
4. 适合现场演示。

## 9. 推荐实施顺序

建议按以下顺序推进：

```text
1. CSV -> JSON 数据转换
2. React + Vite 基础 APP
3. 三个场景切换
4. 轨迹图和状态面板
5. 2D 动画回放
6. 曲线图和 summary
7. 工程交付首页
8. 离线打包
9. Electron 封装
10. 后续 3D 展示
```

不建议一开始就做 3D，因为当前模型是水平面 3-DOF，2D 动画和状态图更能准确表达控制模块的功能。

## 10. 第一版 APP 验收标准

第一版可操作 APP 应满足：

1. 可以在本地浏览器打开；
2. 可以选择直线、圆形、S 形三个演示场景；
3. 可以播放、暂停、重置动画；
4. 可以拖动时间滑条；
5. 可以显示 AUV 位置和航向箭头；
6. 可以显示当前状态数据；
7. 可以显示轨迹、误差、速度、控制输入图；
8. 可以显示项目模块组成；
9. 可以显示控制算法流程；
10. 可以在断网环境下运行；
11. README 中有清晰运行说明。

## 11. 后续增强方向

第一版 APP 稳定后，可继续增强：

1. 增加海流方向可视化；
2. 增加参数扰动场景；
3. 增加更多演示路径；
4. 增加自动生成演示报告；
5. 增加演示视频导出；
6. 增加 Three.js 3D 示意动画；

## 12. 结论

本项目最终展示形式建议采用：

```text
React + Vite + ECharts 可操作 Web APP
```

其优势是：

1. 展示形式更接近正式交付 APP；
2. 可以离线运行；
3. 可以加入动画回放；
4. 可以清楚展示合同中的“控制算法模块及可视化”；
5. 后续可以封装为桌面程序；
6. 后续也可以扩展为 3D 展示。

当前阶段最合适的实施目标是先完成：

```text
离线 2D 可操作 Web APP + 三个典型路径演示 + 动画回放 + 工程交付首页
```

该目标既能满足甲方展示需要，也能保持开发风险可控。
